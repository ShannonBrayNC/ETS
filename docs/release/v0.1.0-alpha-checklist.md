# ETS v0.1.0-alpha Release Checklist

This checklist defines the gates for preparing the ETS `v0.1.0-alpha` release candidate. The alpha is for local validation, protocol review, SDK/API exploration, and controlled demos. It is not production trust infrastructure.

## Release scope

The `v0.1.0-alpha` release should include:

- Canonical JSON hashing and deterministic event hashing.
- `EvidenceEvent` v1 metadata contract.
- Append-only event log with in-memory and SQLite providers.
- Merkle inclusion proofs.
- RC consistency proof behavior with documented limitations.
- Optional Ed25519 tree-head signing.
- FastAPI API under `/api/v1` plus RC compatibility routes.
- Offline verifier CLI and SDK helpers.
- Verification certificate generation.
- Explorer UI for local proof inspection and demo workflows.
- Security, storage, protocol, and release documentation.

## Required validation gates

Run these checks from a clean checkout before tagging:

```powershell
python -m pip install -e ".[dev]"
python -m ruff check .
python -m mypy
python -m pytest
```

If the Explorer UI is present:

```powershell
cd ets/explorer-ui
npm ci
npm run build
```

Security and repository hygiene gates:

- Dependency audit completes or findings are triaged.
- Secret scan completes or findings are triaged.
- No generated fixture contains real credentials, tokens, keys, customer data, private evidence, or personal data.
- `.env.example` uses placeholders only.
- Local development auth remains clearly separated from hosted auth.

## Trust-boundary statements required in release notes

The release notes must state:

- ETS stores metadata and cryptographic hashes by default, not raw evidence bytes.
- Local unsigned tree heads are not production trust anchors.
- Local header/API-key auth modes are for development and demos only.
- Hosted deployments should use production JWT/JWKS validation with issuer and audience checks.
- RC consistency proofs are documented as alpha behavior and should not be described as final production-grade log consistency auditing.
- SQLite is acceptable for local validation and demos, while production storage design remains future work.

## Branch and PR gates

Before tagging:

- The active implementation branch has a reviewed PR path toward `main`.
- Branch divergence has been resolved intentionally.
- CI has passed on the PR or all failures are documented with blocking follow-up issues.
- The PR description includes known limitations and release notes impact.
- The PR has not been auto-merged.

## Tag criteria

The `v0.1.0-alpha` tag may be created only after:

- All required validation gates pass.
- All Sprint 1 gates are complete.
- Any incomplete Sprint 2 through Sprint 4 items are explicitly listed as release limitations or are marked as blocking.
- Release notes are drafted.
- No production deployment is coupled to the tag.

Recommended tag:

```text
v0.1.0-alpha
```

Recommended release title:

```text
ETS v0.1.0-alpha: Local protocol validation and verifier preview
```

## Release notes template

## Summary

Initial alpha release for local ETS protocol validation, API exploration, offline verifier testing, and controlled demos.

- Canonical event hashing
- EvidenceEvent v1 metadata contract
- Append-only log and SQLite demo persistence
- Merkle inclusion proofs

- Not production trust infrastructure
- Local unsigned mode is not a trust anchor
- RC consistency proof behavior is documented alpha behavior
